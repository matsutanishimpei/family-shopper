# セキュリティ診断レポート

現状のアプリケーション構成におけるセキュリティ上のリスクと対策状況をまとめました。

## 1. 脆弱性とリスク

### 🟢 XSS (クロスサイトスクリプティング) [低]
- **現状**: クライアントサイドのJavaScript（`main.js`, `admin.js`）において、ユーザー入力（アイテム名、ユーザー名等）を表示する際に `escapeHTML` 関数によるエスケープ処理を導入しました。
- **リスク**: 基本的には抑えられていますが、将来的に `innerHTML` を多用する新機能を追加する際は注意が必要です。
- **対策**: 引き続きエスケープ処理を徹底し、長期的には `textContent` または React/Hono JSX のサーバーサイドレンダリングを主軸にします。

### 🟡 クッキーの改ざんリスク [中]
- **現状**: 認証状況（`session`）と権限（`role`）、所属（`family_id`）をクッキーに保存していますが、署名（デジタル署名）がされていません。
- **リスク**: 悪意のあるユーザーが `family_id` を書き換えて他家族のデータを覗き見ようとする可能性があります（API側でも特定の検証を行っていますが、クッキー自体の信頼性はまだ改善の余地があります）。
- **対策**: Honoの `signedCookie` への移行を強く推奨します。

### 🟢 SQLインジェクション [解消済み]
- **現状**: 全てのDB操作で `bind()` パラメータを使用しています。
- **リスク**: なし。

### 🔵 マルチテナントにおけるデータ分離 [解消済み]
- **現状**: 全てのデータ取得・更新において `WHERE family_id = ?` によるフィルタリングを徹底しています。
- **リスク**: 実装ミスによるデータ漏洩。
- **対策**: テストコードにより、他家族のIDを指定してもアクセスできないことを定期的に検証します。

### 🟢 シークレット情報の管理 [解消済み]
- **現状**: 環境変数（Secrets）で管理されています。
- **リスク**: なし。

## 2. 実施済みの対策

1. **XSS対策の導入**: `escapeHTML` によるエスケープ処理を全表示箇所に適用しました。
2. **マルチテナント隔離**: APIレベルでの `family_id` による厳格なデータフィルタリング。
3. **パスワードハッシュ化**: `SHA-256` によるハッシュ化保存。

## 3. 推奨される今後の対策

1. **クッキーの署名化**: `session` や `family_id` を改ざん不可能にするため、秘密鍵を用いた署名付きクッキーへ移行する。
2. **ログイン試行制限**: ブルートフォース攻撃を防ぐためのレート制限（Rate Limiting）の導入。
3. **バリデーションの強化**: バリデーションライブラリ（Zod等）を導入し、サーバーサイドでの入力値チェックをより厳格にする。
